# Unified Build Code Policy (synonym: Build ID)

FireDM ships **two** identifiers for every build, both flowing through the entire
release pipeline together:

| Concept | Example | Source of truth |
|---------|---------|-----------------|
| **Product version** | `2022.2.5` | `firedm/version.py::__version__` (also surfaced through `pyproject.toml [tool.setuptools.dynamic].version`). |
| **Build code** *(synonym: build ID)* | `20260428_V1` | Generated by `scripts/release/build_id.py`; the user-facing facade is `scripts/release/versioning.py select-build-code`. |

The product version is stable across many CI runs. The build code is unique
per build attempt, increments on the same calendar day, and survives
re-builds.

## Build code format

```text
YYYYMMDD_V{N}
```

Examples:

```text
20260428_V1
20260428_V2
20260428_V12
```

Validation rules (enforced by `versioning.validate_build_code`):

* the date must parse as a real calendar date (`datetime.strptime(value, "%Y%m%d")`);
* `N` must be a positive integer with no leading zeros;
* the full code matches `(?<!\d)\d{8}_V[1-9]\d*(?!\d)`.

## Same-day enumeration

When you do not pass an explicit code, the next index is the maximum existing
index for that date plus one. The discovery sources are, in order:

1. local artifacts under `dist/` (filename matches and `build_id` keys in JSON manifests);
2. local git tags (`build-*`);
3. (optional) remote git tags via `--include-remote-tags`;
4. (optional) GitHub releases via `--include-github-releases`.

If `V1`, `V2`, and `V5` exist, the next code is `V6`. The script never
overwrites an existing code unless `--allow-overwrite` is passed.

## Cross-platform parity

The same build code is used in a single release run for **every** platform.
`.github/workflows/draft-release.yml` enforces this:

1. `resolve-build-code` (ubuntu-latest) generates one code and exposes it via job outputs (`build_code`, `tag`, `release_name`, `channel`).
2. `build-windows` (windows-latest) and `build-linux` (ubuntu-latest) both consume `needs.resolve-build-code.outputs.build_code`.
3. `release` (ubuntu-latest) downloads both lanes' artifacts, runs `merge_release_manifest.py`, regenerates checksums via `generate_checksums.py`, and dry-runs (or, when explicitly authorized, publishes) the GitHub release.

## Tag / release naming

| Concept | Format |
|---------|--------|
| Git tag | `build-YYYYMMDD_VN` (helper: `versioning.tag_name_for`) |
| Release title | `FireDM YYYYMMDD_VN` (helper: `versioning.release_name_for`) |

## Generated runtime build info

`scripts/release/versioning.py` writes `firedm/_build_info.py` before each
PyInstaller payload build. The module exports:

```python
PRODUCT_VERSION = '2022.2.5'
BUILD_CODE = '20260428_V1'
BUILD_ID = '20260428_V1'
TAG_NAME = 'build-20260428_V1'
RELEASE_NAME = 'FireDM 20260428_V1'
CHANNEL = 'dev'
PLATFORM = 'linux'  # or 'windows'
ARCH = 'x64'
BUILD_INFO = { ... full record ... }
```

The file is generated at packaging time and is git-ignored. PyInstaller picks
it up via the `firedm._build_info` hidden import that both
`scripts/firedm-win.spec` and `scripts/firedm-linux.spec` add when the file
exists.

## Artifact naming

| Lane | Filename pattern |
|------|------------------|
| Windows installer | `FireDM_Setup_<BUILD_CODE>_<CHANNEL>_win_x64.exe` |
| Windows portable | `FireDM_<BUILD_CODE>_<CHANNEL>_win_x64_portable.zip` |
| Linux portable | `FireDM_<BUILD_CODE>_<CHANNEL>_linux_x64.tar.gz` |
| Per-platform manifest | `FireDM_release_manifest_<BUILD_CODE>_<windows|linux>.json` |
| Merged manifest | `FireDM_release_manifest_<BUILD_CODE>.json` |
| Checksum file | `SHA256SUMS_<BUILD_CODE>.txt` |
| License inventory | `license-inventory_<BUILD_CODE>.json` |
| Release notes | `FireDM_release_notes_<BUILD_CODE>.md` |
| Dependency status | `dependency-status_<BUILD_CODE>.json` |

The cross-platform checksum file (`generate_checksums.py`) covers every
artifact listed in the merged manifest. Stale artifacts (different build code)
are rejected by `github_release.py`.

Compatibility aliases remain for tooling that expects stable paths:

```text
dist\release-manifest.json
dist\release-body.md
dist\checksums\SHA256SUMS.txt
dist\licenses\license-inventory.json
```

## CLI quick reference

```text
# Print the next available build code (auto-increment).
python scripts/release/versioning.py select-build-code --json

# Validate an explicit build code.
python scripts/release/versioning.py select-build-code --build-code 20260428_V1 --json

# Read the canonical product version.
python scripts/release/versioning.py product-version

# Write firedm/_build_info.py for a packaged build.
python scripts/release/versioning.py write-build-info \
    --build-code 20260428_V1 --channel dev --platform linux --arch x64
```

Local Windows builds:

```powershell
.\build-release.bat dev
.\.venv\Scripts\python.exe scripts\release\build_windows.py --arch x64 --channel dev --date 20260427
.\build-release.bat dev --build-id 20260427_V7
```

Local Linux builds (Linux host or WSL only):

```bash
bash scripts/linux-build.sh --channel dev --arch x64
bash scripts/linux-build.sh --channel dev --arch x64 --build-code 20260428_V1
python scripts/release/build_linux.py --arch x64 --channel dev --build-id 20260428_V1
```

Explicit reuse is blocked unless `--allow-overwrite` is also provided. Do not
use overwrite for normal release creation.

## Why "build code" vs "build ID"

External automation often uses *build code*; the FireDM repository historically
uses *build_id*. The two are exact synonyms — every manifest carries both
`build_id` and `build_code` fields, the workflow input is `build_id`, and
`versioning.py` exposes the `build_code` API. Renaming the wire field would
break existing draft releases, so we publish both names.

## Blocked lanes

Blocked until implemented and validated: x86, ARM64, universal bootstrapper,
MSI, MSIX, AppImage, .deb/.rpm, FFmpeg/ffprobe bundling. Stable channel
publishing additionally requires signed installer artifacts.
